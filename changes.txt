PHASE 3: SMOTE CLASS BALANCING
Section: "PREPARING DATA WITH SMOTE"

PROCESS:
1. Extract features (X) and labels (y)
2. Stratified train/test split (80/20)
3. Scale features using StandardScaler (IMPORTANT: scale before SMOTE)
4. Apply SMOTE:
   - sampling_strategy=0.30 (oversample fraud to 30% of majority)
   - Generates synthetic fraud examples
   - Trains model on 70% legit, 30% fraud

BEFORE SMOTE:
- Fraud: 0.2% of data
- Non-fraud: 99.8% of data
- Model learns to ignore fraud signals

AFTER SMOTE:
- Fraud: 30% of training data
- Non-fraud: 70% of training data
- Model learns fraud patterns effectively

WHY 30% NOT 50%:
- 50% would be unrealistic (real data is 0.2%)
- Model would overfit to synthetic data
- 30% is sweet spot: learns patterns without overfitting

TIME: ~5-10 minutes

PHASE 7: ENSEMBLE VOTING
Section: "ENSEMBLE VOTING (Weighted Average)"

CONCEPT:
- Combine predictions from 3 models
- Weight by individual performance
- Reduces individual model weaknesses

FORMULA:
final_probability = (
    0.45 * xgb_proba +      # XGBoost weight
    0.40 * lgb_proba +      # LightGBM weight

WHY THESE WEIGHTS:
- XGBoost: 45% (most reliable single model)

BENEFITS:
- No single model dominates
- Reduces overfitting
- More robust to adversarial fraud patterns
- Example: XGBoost misses fraud X, but LGB catches it

OUTPUT:

TIME: Negligible (just arithmetic)


PHASE 8: OPTIMAL THRESHOLD FINDING
Section: "FINDING OPTIMAL THRESHOLDS"

BACKGROUND:
- Models output probability (0-1)
- Decision threshold transforms to binary (fraud/legit)
- Default threshold: 0.5
- Problem: Often suboptimal for imbalanced data

PRECISION-RECALL CURVE:
- Trades off precision vs recall
- 0% recall: No alerts (missing fraud)
- 100% recall: All transactions flagged (no useful)
- Sweet spot: ~75% recall with highest precision

APPROACH:
1. Calculate precision-recall curve
2. Find threshold for 75% recall (catch 3/4 frauds)
3. Find threshold for max F1 (balance precision/recall)
4. Use F1-optimized threshold (better for imbalanced data)

RATIONALE:
- 75% recall: Catch most fraud, acceptable false negatives
- High precision: Don't cry wolf too much
- F1 maximization: Optimal balance

OUTPUTS:
- threshold_75_recall: For monitoring
- threshold_f1: For production
- optimal_threshold: Selected threshold
- Example: 0.35-0.45 (much lower than 0.5!)

TIME: ~2-3 seconds


PHASE 9: EVALUATION & METRICS
Section: "FINAL ENSEMBLE PERFORMANCE"

METRICS CALCULATED:

1. CONFUSION MATRIX:
   TN | FP
   FN | TP
   
   - TN: Correctly identified legit transactions
   - FP: False positives (normal but flagged)
   - FN: False negatives (fraud but missed)
   - TP: True positives (correctly caught fraud)

2. PRECISION = TP / (TP + FP)
   - Of all flagged transactions, how many are actually fraud?
   - Example: 75% precision = 3 out of 4 alerts are real fraud

3. RECALL = TP / (TP + FN)
   - Of all fraud cases, how many did we catch?
   - Example: 75% recall = catching 3 out of 4 frauds

4. F1 SCORE = 2 * (precision * recall) / (precision + recall)
   - Harmonic mean of precision/recall
   - Single metric for model quality
   - Example: 0.75 F1 score = well-balanced model

5. ROC-AUC = Area Under the Curve
   - Probability that model ranks random fraud > random legit
   - 0.5 = random guessing
   - 1.0 = perfect model
   - 0.85 = good model
   - 0.88+ = industry standard

PRINTED OUTPUT:
- Classification report (precision/recall/f1 for both classes)
- Summary metrics
- Confusion matrix
- Top 15 feature importances

Problem: OutOfMemory error

```python
df = load_and_preprocess(sample_size=50000)  # Instead of 500k
```

Problem: SMOTE is slow
Solution: Normal, it's generating synthetic data
- With 10k fraud cases, generates 3k synthetic (slow)
- Can reduce: sampling_strategy=0.10 instead of 0.30

Problem: LightGBM fails with categorical features
Solution: Already handled in train_model_v3.py
- Creates encoded columns (numeric) before training

Problem: Training is getting stuck
Solution: Check eval_set is not same as training set
- Should be: eval_set=[(X_test_scaled, y_test)]

Problem: Models too large
Solution: Reduce n_estimators
```python
n_estimators=200  # Instead of 500
```
