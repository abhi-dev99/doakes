# Documentation: `explainable_ai.py`

## 1. Purpose
This module implements the **Explainable AI (XAI)** layer for the ARGUS platform. It translates opaque machine learning probabilities (e.g., "Risk: 0.82") into human-readable, actionable sentences (e.g., "Transaction amount is 5x higher than user's 30-day average at an unusual 2:00 AM hour"). 

## 2. SHAP vs Heuristics (Design Decision)
*The most critical architectural decision in this file is the explicit rejection of the SHAP (SHapley Additive exPlanations) library.*
While SHAP is the industry standard for ML explainability, computing Shapley values for XGBoost/LightGBM ensembles requires significant matrix multiplications. Tests showed SHAP added 50-100ms of latency per transaction.
Since ARGUS requires sub-20ms latency to act as a *Pre-Authorization* blocking engine, this module uses a **Heuristic Weight-Based Mapper** instead.

## 3. Core Logic / Workflow (`FraudExplainer`)

### A. Predefined Weights (`FEATURE_WEIGHTS`)
The class maintains a static dictionary mapping raw feature names to human-readable strings and base "importance weights" derived from offline SHAP analysis.
Examples:
*   `amount_zscore`: ("Unusually high amount for this user", 0.85)
*   `is_night`: ("Transaction occurred during high-risk night hours", 0.60)
*   `is_new_device`: ("First time using this device", 0.75)

### B. Explanation Generation (`generate_explanation`)
When a transaction is scored, the `FraudExplainer` looks at the highest risk components (returned by the `analyze_transaction` payload).
1.  **Rule Triggers**: It prioritizes hard rules first (e.g., if `UPIFraudDetector` flagged a Digital Arrest, that becomes the primary explanation).
2.  **Dynamic Anomalies**: It reads the `behavior_analysis` dict. If `amount_zscore` > 3, it dynamically generates the string: "Amount is significantly higher than user's normal pattern."
3.  **Feature Mapping**: For remaining risk, it maps the triggered features to their human-readable strings from `FEATURE_WEIGHTS`.

### C. Persona Formatting
The explainer formats the output into bullet points designed specifically for the Dashboard UI, prioritizing the most heavily weighted features at the top of the list.

## 4. Assumptions & Limitations
*   **Limitation (Approximation)**: Because this is a heuristic mapper and not true SHAP, it only *approximates* why the model made a decision. It maps features that are generally highly weighted by XGBoost, rather than calculating the exact mathematical contribution of a feature to a specific inference.
*   **Assumption**: Assumes the `behavior_analysis` and `triggered_rules` arrays passed from `fraud_model.py` are accurately populated.

## 5. Role in Architecture
This module sits right before the API response is sent back to the client. It takes the raw JSON numbers generated by `fraud_model.py` and enriches them with the `explanation_summary` array that is rendered in the ARGUS Dashboard for human analysts.
